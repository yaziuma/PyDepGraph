### LLMが利用することを想定した改善案の概要

`pydepgraph`をLLMが利用する際の改善策として、メタデータの厳格化と、CLIを活かした自然言語クエリの代替案について、以下の概要を提案します。

---

### 1. メタデータフィールドの標準化と提案

LLMがコードの役割を正確に理解できるよう、特定の役割を示すメタデータフィールドを標準化します。これにより、LLMは自由形式のテキストに頼るのではなく、構造化されたデータとしてコードの役割を認識できます。

* **`role`フィールドの導入**: `Module`, `Function`, `Class`モデルに、`role: Optional[str]`というフィールドを新設します。このフィールドは、コードの役割を分類する目的で使用します。

* **推奨される`role`の値**: 以下のような固定された、あるいはセミ固定的な値のリストを定義し、プロジェクトのコーディング規約や設定ファイルを通じて適用することを推奨します。

    * **モジュール**:
        * `api`: 外部に公開するインターフェース
        * `service`: ビジネスロジックを担うサービスレイヤー
        * `data_access`: データベースや外部APIとの通信
        * `model`: データ構造の定義
        * `cli`: コマンドラインインターフェース
        * `util`: 汎用的なユーティリティ関数やクラス
        * `config`: 設定情報

    * **関数/クラス**:
        * `public_api`: 外部から利用されることを想定した公開API
        * `private`: 内部利用に限定された非公開のロジック
        * `entrypoint`: プログラムのエントリーポイント（`main`関数など）
        * `factory`: オブジェクトの生成を専門とするクラスや関数
        * `test`: テストコード

---

### 2. LLMフレンドリーな「プリセット検索」コマンドの拡充

コストのかかる自然言語解析を避けつつ、LLMが多様なクエリを実行できるように、CLIコマンドとして実行可能な「プリセット検索」機能を拡充します。これにより、LLMは特定の分析タスクをコマンドとして呼び出すだけで、意図した結果を得ることができます。

* **`pydepgraph query` コマンドの強化**: 既存の`query`コマンドに、以下のような新しいサブコマンドやオプションを追加します。

    * **`pydepgraph query role --type <node_type> --value <role_value>`**:
        * 特定の役割を持つノード（モジュール、関数、クラス）を検索します。
        * 例: `pydepgraph query role --type module --value api`
        * これにより、LLMは「APIレイヤーを構成するモジュールはどれか？」といった質問に対する答えを直接取得できます。

    * **`pydepgraph query dependents --target <node_name>`**:
        * 特定のノードに依存しているすべてのノードを検索します。
        * 例: `pydepgraph query dependents --target src/pydepgraph/database.py`
        * 「`database.py`を変更した場合、影響を受けるファイルはどれか？」といった分析に役立ちます。

    * **`pydepgraph query complex-functions --min-complexity <value>`**:
        * 循環複雑度が特定の値を上回る関数を検索します。
        * 例: `pydepgraph query complex-functions --min-complexity 15`
        * これにより、LLMは保守性の低い関数やリファクタリング候補を特定できます。

これらの改善により、LLMは定められたルールとコマンド体系に従うことで、コストを抑えつつも高度な分析を動的に実行できるようになります。
