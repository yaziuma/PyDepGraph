# 詳細設計: メトリクスと中心性分析 (v2)

## ⚠️ 重要な注意事項

**このドキュメントの修正履歴**:
- 既存GraphAnalyticsServiceとの統合を明記
- GraphConverter → 既存NetworkX統合の拡張に修正済み
- LCOM4は複雑性のため優先度低に調整

## 1. 概要

本ドキュメントでは、生成された依存関係グラフを基に、コードの構造的な品質を評価するためのメトリクス計算機能と、グラフ理論に基づいた中心性分析機能に関する詳細設計を定義する。これにより、保守性の低いコードやアーキテクチャ上の弱点を客観的な指標に基づいて特定する。

## 2. 設計詳細

### 2.1. 担当クラス

-   `pydepgraph.services.graph_analytics_service.GraphAnalyticsService` (既存のものを拡張)

### 2.2. NetworkX統合の拡張

`GraphAnalyticsService`には既に`_build_graph()`メソッドでNetworkX統合が実装されている。この既存機能を拡張してより高度なメトリクス計算を行う。

### 2.3. メトリクスの拡張

#### 2.3.1. ファンイン・ファンアウト

-   **定義**:
    -   **ファンイン**: あるノードに向かうエッジの数。そのモジュールがどれだけ「利用されているか」を示す。
    -   **ファンアウト**: あるノードから出るエッジの数。そのモジュールがどれだけ「他に依存しているか」を示す。
-   **計算方法**:
    -   既存の`_build_graph()`で生成されたNetworkXグラフで、`DiGraph.in_degree()`と`DiGraph.out_degree()`メソッドを呼び出すだけで計算できる。
-   **実装**: `GraphAnalyticsService`内に`calculate_fan_in_out()`メソッドを実装する。

#### 2.3.2. 凝集度 (LCOM4)

-   **定義 (LCOM4 - Lack of Cohesion in Methods)**: クラス内のメソッドが、共通のインスタンス変数を介してどれだけ関連しているかを示す指標。値が大きいほど凝集度が低い（＝関連性の低いメソッドが混在している）。LCOM4は、メソッドを「共通のインスタンス変数を直接または間接的に共有するグループ」に分けたときの、グループの数を表す。
-   **計算方法**:
    1.  対象となるクラスのソースコードをASTでパースする。
    2.  **ステップA: 変数アクセスの収集**:
        -   クラス内の各メソッド (`ast.FunctionDef`) について、アクセスしているインスタンス変数 (`self.xxx`) のリストを作成する。これは、`ast.Attribute`ノードで`value`が`ast.Name`かつ`id`が`self`であるものを探すことで実現できる。
    3.  **ステップB: メソッド関連グラフの構築**:
        -   クラス内のメソッドをノードとする無向グラフを構築する。
        -   2つのメソッドが同じインスタンス変数を1つでも共有していれば、それらの間にエッジを追加する。
    4.  **ステップC: 連結成分の計算**:
        -   ステップBで構築したグラフの連結成分 (connected components) の数を数える。この数がLCOM4の値となる。
-   **実装**: `GraphAnalyticsService`内に`calculate_lcom4(class_node)`メソッドを実装する。この処理は複雑なため、独立したヘルパークラスに切り出すことも検討する。

### 2.4. 中心性分析の多様化

`networkx`を利用して、以下の中心性指標を計算する。

#### 2.4.1. 媒介中心性 (Betweenness Centrality)

-   **定義**: あるノードが、他のノードペア間の最短経路上に位置する頻度。システムの「仲介役」や「ボトルネック」を特定するのに役立つ。
-   **計算**: `networkx.betweenness_centrality(nx_graph)`

#### 2.4.2. 近接中心性 (Closeness Centrality)

-   **定義**: あるノードから、グラフ内の他の全ノードへの最短経路長の平均の逆数。他のモジュール全体に影響を及ぼしやすい「中心的な」モジュールを特定するのに役立つ。
-   **計算**: `networkx.closeness_centrality(nx_graph)`

### 2.5. 結果の統合とレポーティング

-   **データ格納**: 計算したメトリクス（ファンイン/アウト、LCOM4、中心性スコア）は、`networkx`グラフのノード属性として追加・更新する。
-   **CLI**:
    -   `pydepgraph report` コマンドに `--metrics` フラグを追加する。
    -   このフラグが指定された場合、通常の依存関係リストに加え、各モジュール/クラスのメトリクスを表形式で出力する。
    -   `--sort-by <metric_name>` のようなオプションで、特定指標の高い順にソートして表示できるようにする。
-   **レポート例**:
    ```
    $ pydepgraph report --metrics --sort-by betweenness

    📊 Module Metrics & Centrality
    ┌──────────────────────────────────┬─────────┬──────────┬─────────────┬───────────┐
    ```

## 3. 期待される効果

-   **リファクタリング対象の明確化**: 結合度が高く（ファンイン/アウト大）、凝集度が低い（LCOM4大）クラスや、異常に中心性が高いモジュールを特定し、優先的に改善対象とすることができる。
-   **設計レビューの高度化**: 「このモジュールの中心性が高すぎるため、責務を分割すべき」といった、データに基づいた具体的な設計議論が可能になる。
-   **潜在的リスクの発見**: システムのボトルネックとなりうるモジュールを事前に特定し、重点的なテストやドキュメント整備の対象とすることができる。
