> **外部ライブラリ依存関係の抽出 (優先度: 中)**:
- requirements.txtやpyproject.tomlをパースして外部ライブラリ依存を抽出
- 現在のCode2Flowは既にファイル横断分析が可能のため、この機能は重複の可能性あり
- SBOM生成や脆弱性管理には有効

**注意**: ファイル横断的関数呼び出し分析は、既存のCode2FlowExtractorが実装済みのため不要

優先度中

> 依存関係の進化分析: pydepgraph/incremental.pyに増分分析のプレースホルダーコードがあるものの、機能は未実装です。過去の分析結果と現在の結果を比較し、モジュールの結合度が時間の経過とともにどのように変化したかを追跡することで、コードベースの健全性トレンドを可視化できます。これにより、問題が潜在化する前に早期発見できるようになります。

優先度中

> メトリクスの拡張: モジュールやクラスの凝集度、結合度、ファンイン・ファンアウトといったメトリクスを計算し、高密度で保守が困難な部分（"God Object"など）を特定できるようにします。
> 中心性分析の多様化: PageRank以外に、媒介中心性や近接中心性などのアルゴリズムを導入し、依存関係のハブとなっているモジュールや、他のモジュールへの影響力が高いモジュールを多角的に評価できるようにします。

**実装可能性による優先度調整**:
- **ファンイン・ファンアウト、基本中心性**: 優先度中 (NetworkX既存機能活用)
- **LCOM4**: 優先度低 (複雑なAST解析と計算が必要)
- **高度な中心性分析**: 優先度中 (NetworkX既存機能活用)


## Geminiとの議論結果

  Geminiとの対話を完了しました。以下が主要なレビュー結果と推奨事項です：

  📊 総合評価

  3つの改善提案は全体的に優秀で技術的に堅実。PyDepGraphの論理的な次のステップとして適切で、既存アーキテクチャとの整
  合性も高い。

  🎯 実装優先度の推奨順序

  1. 提案1（依存関係分析強化） - 最高優先度
  2. 提案3（メトリクス分析） - 中優先度
  3. 提案2（進化分析） - 低優先度

  理由: コアデータ品質の向上が最優先。メトリクスは低労力・高価値。進化分析は高度だが複雑。

  🔍 技術的詳細レビュー

  提案1: 依存関係分析強化

  - 技術精度: 高 - 2パス戦略は標準的で堅実
  - アーキテクチャ整合性: 高 - DependencyFileExtractor追加は既存パターンに適合
  - 実装複雑度: 中 - クロスファイル解析は状態管理が必要
  - スキーマ影響: 軽微 - is_externalフラグ活用を継続推奨

  提案2: 進化分析

  - 技術精度: 高 - Git連携とスナップショット比較は合理的
  - アーキテクチャ整合性: 高 - IncrementalAnalyzer拡張は適切
  - 実装複雑度: 中-高 - GraphComparatorの差分定義が複雑
  - ストレージ: JSON → MessagePack → Protocol Buffersの段階的移行推奨

  提案3: メトリクス分析

  - 技術精度: 高（LCOM4は条件付き）
  - アーキテクチャ整合性: 高 - GraphAnalyticsService拡張は正解
  - 実装複雑度: 低-中 - NetworkXとの連携が鍵
  - パフォーマンス: Kùzu（単純メトリクス）+ NetworkX（複雑アルゴリズム）のハイブリッド推奨

  💡 重要な技術推奨事項

  1. LCOM4実装: Code2FlowExtractorのAST解析を拡張し、visit_Attributeメソッドでインスタンス変数アクセスを捕捉
  2. パフォーマンス最適化:
  大規模プロジェクト（10k+ファイル）向けに増分インデックス化とSQLite/LevelDBベースのディスク索引
  3. メトリクス計算: 単純メトリクス（次数）はCypherクエリ、複雑アルゴリズム（中心性）はNetworkX利用
  4. テスト戦略: 対象特化統合テスト + スナップショット（Golden）テストの組み合わせ

  🎉 結論

  全提案が実装価値があり、PyDepGraphを実用的で強力な依存関係分析ツールに発展させる明確な道筋を提供している。特に提
  案1の実装により、ツールのコアデータ品質が大幅に向上し、後続の機能拡張の基盤が強固になります。
